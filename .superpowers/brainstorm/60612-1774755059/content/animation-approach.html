How should layered animation work?

Current system: 7x3 spritesheet grid (walk, type, read x 3 directions). Each layer needs to animate in sync. This is the hardest technical decision.

A

Stacked Spritesheets

Each layer (hair, torso, legs...) has its own 7x3 spritesheet, same frame layout as today. At runtime, 5 sprites play the same animation in sync, stacked on top of each other.

Pros: Simple rendering, uses existing Phaser animation system
Cons: Every layer variant needs a full spritesheet (hair_01.png = 7x3 grid = 21 frames)
B

Runtime Compositing (Canvas)

Compose all layers into a single spritesheet at character creation time using OffscreenCanvas. Result is one traditional spritesheet per character — identical to current system at runtime.

Pros: Zero runtime overhead after compositing, exact same rendering path as today
Cons: Compositing step on load, still need aligned spritesheets per layer
C

Skeletal / Bone Animation

Each part is a separate static image positioned by bones. Animations move bones, parts follow. Like paper dolls with joints.

Pros: Fewest assets needed, most flexible
Cons: Doesn't look like pixel art (parts rotate/scale smoothly), needs animation editor, major engine change

Recommendation: B (Runtime Compositing)

Keeps the existing animation system untouched. Layers are composed into a single spritesheet once (on character creation or load), then the game runs exactly as it does today. Best balance of flexibility vs. engine complexity.
